# PEZY-DPM user guide

This guide describes the packaged input data, program output, and the shortest
commands for reproducing CPU MPI and MPI + PEZY-SC3s smoke tests. For the
original DPM physics workflow, geometry format, preprocessing programs, and
historical sample problem, also consult [`readme.txt`](readme.txt).

PEZY-DPM is research software. It is not approved for clinical treatment
planning, medical diagnosis, or patient-specific clinical use.

## 1. Execution environments

The CPU MPI target can be built and run on a conventional CPU node with GNU
C/C++ and Fortran compilers and an MPI implementation. It does not require
PEZY hardware or PZSDK/PZCL.

The MPI + PEZY-SC3s target requires PEZY-SC3s hardware, PZSDK/PZCL, and MPI.
The results reported in the associated manuscript use this target. Unless a
command says otherwise, run all examples below from the repository root;
paths embedded in the input files are relative to that directory.

## 2. Build the programs

### CPU MPI

```bash
make clean MODE=mpi
make mpi PRECISION=float
```

This creates `dpm.mpi`. Use `PRECISION=double` to select double precision for
the C++ transport implementation.

### MPI + PEZY-SC3s

Set the local PZSDK and MPI paths first. The following values are examples and
must be adapted to the installation:

```bash
export PATH=/usr/local/mpich/bin:/opt/pezy/pzsdk/bin:$PATH
export LD_LIBRARY_PATH=/usr/local/mpich/lib:/opt/pezy/pzsdk/lib:$LD_LIBRARY_PATH
export PZSDK_PATH=/opt/pezy/pzsdk
export PZCLKernelTimeout=1000000000
```

Build the final four-active-thread kernel variant used for the manuscript:

```bash
make clean MODE=mpi_sc3s
make mpi_sc3s PRECISION=float KERNEL_VERSION=4th PZC_TARGET_ARCH=sc3s
```

This creates the host executable `dpm.sc_mpi` and the device binary
`kernel/kernel.pz`. The repository also supports `KERNEL_VERSION=8th` and
`PRECISION=double` for implementation testing.

## 3. DPM input-file fields

Each input value follows its descriptive line. The supplied input files use
the following fields in this order:

| Field | Meaning |
| --- | --- |
| `No of histories` | Number of source-particle histories to simulate. |
| `Allotted time` | Time limit in seconds. A negative value denotes CPU time and a positive value denotes elapsed time in the original DPM convention. If the platform time routines are disabled, termination is controlled by the history count. |
| `Particle Type` | `-1` for electrons and `0` for photons. |
| `Source energy` | Monoenergetic source energy in eV. |
| `Square source beam size` | Side length of the square field in cm; `0` selects a pencil beam. |
| `Electron absorption energy` | Electron transport cutoff in eV. |
| `Photon absorption energy` | Photon transport cutoff in eV. |
| `Prefix of all compatible files` | Common prefix of the physics tables produced by PREDPM. Electron and photon cases use different table sets. |
| `Geometry file to use` | Path to the DPM voxel geometry. |
| `Rectangular Region of Interest` | Initial and final coordinates for x, y, and z in cm. Dose is reported for this region. |
| `Random seeds` | The original DPM source-seed pair. `0 0` selects the default initialization. Parallel worker substreams are initialized from the packaged files in `data/seeds/`. |

The packaged smoke inputs differ from the full benchmark inputs only in the
number of histories:

| Case | Smoke input | Full benchmark input |
| --- | --- | --- |
| 20 MeV electron | `data/input/electron_20MeV_smoke1e4.in` | `data/input/electron_20MeV_2p5e8.in` |
| 6 MeV photon | `data/input/photon_6MeV_smoke1e4.in` | `data/input/photon_6MeV_2p5e8.in` |

## 4. Runtime data directories

- `data/pre/` contains preprocessed electron and photon physics tables read by
  DPM. The `pre4elec.*` and `pre4phot.*` groups were generated by the original
  PREDPM workflow for the corresponding transport settings.
- `data/vox/` contains voxel geometries. A `.vox` file specifies voxel counts,
  voxel dimensions, and the material number and mass density of each voxel.
- `data/seeds/` contains pre-generated RANECU seed-component files used to
  initialize separated random-number substreams for parallel workers. The
  number of available seed pairs must cover the workers used by a run.
- `data/output/` receives redirected standard output, progress reports, and
  smoke-test logs. Generated outputs are ignored by Git except for `.gitkeep`.
- `data/pendat/` and `data/benchmarks/` preserve material and benchmark data
  inherited from the original DPM workflow.

## 5. Minimal reproducibility tests

The two scripts below run both named cases, verify that exactly 10,000
histories were reported, and check for the normal `Have a nice day.` DPM
termination message.

### CPU MPI smoke test

```bash
make clean MODE=mpi
make mpi PRECISION=float
MPI_RANKS=2 bash run/smoke_cpu_mpi.sh
```

Run only one case by passing `electron` or `photon`:

```bash
MPI_RANKS=2 bash run/smoke_cpu_mpi.sh electron
```

### MPI + PEZY-SC3s smoke test

```bash
make clean MODE=mpi_sc3s
make mpi_sc3s PRECISION=float KERNEL_VERSION=4th PZC_TARGET_ARCH=sc3s
MPI_RANKS=1 bash run/smoke_pezy_sc3s.sh
```

One MPI rank controls one PEZY-SC3s device. For a two-device smoke run, use:

```bash
MPI_RANKS=2 bash run/smoke_pezy_sc3s.sh
```

The scripts write the following files:

```text
data/output/cpu_mpi_electron_smoke.out
data/output/cpu_mpi_photon_smoke.out
data/output/pezy_sc3s_electron_smoke.out
data/output/pezy_sc3s_photon_smoke.out
```

The 10,000-history inputs are intended only to verify build, launch, data-file
access, and normal termination. They are too small for dosimetric or
performance conclusions.

## 6. Full benchmark commands

After the smoke tests pass, the manuscript-scale inputs can be launched with
the same binaries. For example:

```bash
mpirun -n 64 ./dpm.mpi \
  < data/input/photon_6MeV_2p5e8.in \
  > data/output/mpi64_photon_2p5e8.out

mpirun -n 1 ./dpm.sc_mpi \
  < data/input/electron_20MeV_2p5e8.in \
  > data/output/pezy1_electron_2p5e8.out
```

Select the rank count according to the available CPU cores or PEZY-SC3s
devices. Full benchmarks can take substantially longer than the smoke tests.

## 7. Output and dose results

The main result is written to standard output. Redirect it to a file as shown
above. The final report contains:

- an echoed summary of the source, cutoffs, physics tables, geometry, and ROI;
- a voxel-dose table with columns `x(cm)`, `y(cm)`, `z(cm)`, `dose(MeV/g)`,
  and `+-`;
- the number of simulated histories;
- internal timing and efficiency statistics when the platform timing routines
  are available; and
- the normal termination message `Have a nice day.`

The dose table is normalized by the number of source histories and reports the
scored energy deposition per voxel mass. The `+-` column reports a statistical uncertainty equal to twice the standard deviation. A command-file request can
also write an intermediate report to `data/output/progress.out`.

## 8. Basic result checks and post-processing

Confirm the history count and successful completion:

```bash
grep -A1 "No of histories simulated:" data/output/cpu_mpi_electron_smoke.out
grep "Have a nice day." data/output/cpu_mpi_electron_smoke.out
```

Extract the final dose rows into a whitespace-separated table:

```bash
awk '
  /x\(cm\).*dose\(MeV\/g\)/ {dose=1; next}
  dose && /report: Performance statistics:/ {exit}
  dose && NF==5 && $1 ~ /^[-+0-9.]/ {print $1, $2, $3, $4, $5}
' data/output/cpu_mpi_electron_smoke.out > data/output/electron_dose.dat
```

For correctness studies, compare matching voxel coordinates and normalized
dose values between implementations generated from the same input. The smoke
tests check software execution only; use statistically adequate history counts
and an explicitly defined dose-comparison metric for scientific validation.

## 9. Troubleshooting

- Run commands from the repository root. The packaged input files contain
  repository-relative paths such as `data/pre/...` and `data/vox/...`.
- If `mpicxx` or `mpif90` is missing, load the MPI environment before building.
- If a PEZY build cannot find PZCL headers, libraries, or tools, verify
  `PZSDK_PATH`, `PATH`, and `LD_LIBRARY_PATH`.
- If a parallel run exhausts the seed input, reduce the worker count or provide
  a compatible, documented RANECU seed set with enough seed pairs.
- See [`../TESTED_ON_100.md`](../TESTED_ON_100.md) for the recorded build and
  smoke-test matrix on server100.
